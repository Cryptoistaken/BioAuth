Protocol

Enrollment

  1. Client calls navigator.credentials.create()
  2. Device triggers platform authenticator (fingerprint)
  3. Client receives credential.rawId
  4. Client runs HKDF-SHA256 over rawId with fixed salt + info
  5. 256-bit seed stored locally as identity public key
  6. Optional: POST /enroll with { userId, publicKey, deviceId }

Authentication

  1. Client fetches challenge from server or generates locally
  2. Client calls navigator.credentials.get() with stored credentialId
  3. Device triggers platform authenticator (fingerprint)
  4. If server: POST /verify with { userId, challenge }
  5. Server checks challenge was issued, not expired, not replayed
  6. Returns { verified: true, publicKey }

Honest limitations

Cross-sensor variance

Different hardware sensors produce different raw data from the same finger. V1 works across your own enrolled devices. True device independence is a V2 goal.

PIN fallback

WebAuthn allows PIN/pattern as fallback — the OS controls this. Fingerprint-only enforcement requires a native app using Android BiometricPrompt.

Fingerprint is irrevocable

Unlike a password, you can't change your fingerprint. Liveness detection and multi-factor enrollment are recommended for high-security use cases.
